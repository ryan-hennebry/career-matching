## What Failed

### Failure 1: Forgot to Use JobSpy Until User Reminded
- **What happened:** Agent completed onboarding and was about to start searching, but the user had to ask: "aren't you meant to check jobspy as well?" Agent then realised the subagent template uses `jobspy_search.py` as step 1.
- **Root cause:** Parent orchestrator didn't read the subagent template before dispatching. It jumped to dispatching without reviewing the workflow it was supposed to follow.
- **Principle violated:** Core Rule 3 — MUST search ALL target role types (the search workflow itself was nearly skipped)
- **Fix type:** Implementation — CLAUDE.md wording

### Failure 2: Score Math Not Shown in Presentation
- **What happened:** Results presented with scores (e.g., "Score: 88") but without the math breakdown. The design spec requires: "Show math breakdown for each score" and "Present each job with: score with full breakdown." The score breakdowns exist in the verified JSON files but weren't surfaced to the user.
- **Root cause:** Parent orchestrator summarised results rather than extracting full breakdowns from verified JSON
- **Principle violated:** Presentation Workflow precondition 4 — "Show math breakdown for each score"
- **Fix type:** Implementation — CLAUDE.md presentation section needs stronger wording

### Failure 3: No Cross-Role-Type Deduplication
- **What happened:** BlakBear "Junior Marketing & Operations Associate" appears in both Product Marketing Associate (score 95) and Growth Marketing Associate (score 87) results. DIA "Growth Marketing Associate" appears in both Marketing Associate and Growth Marketing Associate. No dedup was performed.
- **Root cause:** Orchestration Workflow step 8 (cross-role-type deduplication) was not executed
- **Principle violated:** Orchestration Workflow step 8
- **Fix type:** Implementation — parent CLAUDE.md needs to emphasise dedup before presentation

### Failure 4: Inconsistent Presentation Format
- **What happened:** Marketing Manager results presented as a flat list with dashes. Community Manager results presented as a table. Founder's Associate as a table. Marketing Associate as a flat list. Growth Marketing Associate as a table. No consistent format across role types.
- **Root cause:** No format specification in the presentation workflow
- **Principle violated:** UX consistency
- **Fix type:** Implementation — add a standard presentation format to CLAUDE.md

### Failure 5: amicable Job Scored 0 But Still Presented
- **What happened:** Founder's Associate list includes "amicable — Founder's Associate" with score 0 and status "unverified". A score of 0 should mean the job doesn't match at all. Presenting it wastes user attention.
- **Root cause:** No minimum score threshold for presentation
- **Principle violated:** "Recommend, don't list" — showing a 0-score job is noise
- **Fix type:** Implementation — add minimum presentation threshold (e.g., 50+) to CLAUDE.md

### Failure 6: Context.md Sources Not Checked Before Searching
- **What happened:** User had to remind agent that jobspy should be used. This suggests the agent didn't read the subagent template or context.md Sources section before launching searches.
- **Root cause:** Same as Failure 1 — agent dispatched without reviewing its own workflow
- **Principle violated:** Orchestration Workflow steps 2-3 (read all role types, read subagent template)
- **Fix type:** Implementation — same fix as Failure 1

### Failure 7: No Checkpoint After Every 2 Role Types
- **What happened:** Session state was only written once, after all searches completed. The design spec says: "Deterministic checkpoint: Write session-state.md after every 2 role types reach verified status."
- **Root cause:** Agent treated session-state.md as a final summary rather than a progressive checkpoint
- **Principle violated:** Session Management — deterministic checkpoint rule
- **Fix type:** Implementation — strengthen CLAUDE.md checkpoint wording

### Failure 8: 45% Context Used Despite Subagents
- **What happened:** User noted 45% context consumed by the time batch 2 was dispatching. While this didn't cause a failure (session completed), it's higher than necessary.
- **Root cause:** Parent context consumed by: (1) large subagent template prompts being assembled inline, (2) reading verified JSON files to present results, (3) rewriting the full digest with hyperlinks. The agent acknowledged this: "The main context drain is likely the large subagent template prompts being assembled inline."
- **Principle violated:** Core Rule 6 — batch work within context limits
- **Fix type:** Implementation — reduce parent context consumption:
  - Don't inline full templates in conversation — use a more concise dispatch pattern
  - After presenting a role type, drop details per progressive offloading rules
  - Consider having digest generation as a subagent too

## Specific Fixes

### CLAUDE.md Changes

```diff
 ## ORCHESTRATION WORKFLOW

 This is the core search-verify-present loop. All search and verification work is dispatched to subagents.

 1. **Capture run date:** Execute `date +%Y-%m-%d` once.
-2. **Read ALL role types** from `## Target` in context.md
-3. **Read subagent template:** Read `references/subagent-search-verify.md`
+2. **Read ALL role types** from `## Target` in context.md
+3. **Read subagent template:** Read `references/subagent-search-verify.md` — review the full Search Workflow and Verification Workflow sections before dispatching ANY subagents. Confirm you understand the tools each subagent will use (jobspy_search.py, filter_jobs.py, summarize_jobs.py, WebFetch for specialty sources).
```

```diff
 ## PRESENTATION WORKFLOW

 Before presenting ANY role type's results, ALL of these must be true:

 1. All jobs for that role type are verified
 2. Every job has: exact title, working URL, confirmed active status, scored with math
 3. Jobs ranked by score, highest first
-4. Present each job with: title, company, location, salary (if available), score with full breakdown, requirements met/gaps, application URL
+4. Present each job with: title, company, location, salary (if available), score with full breakdown, requirements met/gaps, application URL
+
+**Minimum score threshold:** Only present jobs scoring 50 or above. Jobs below 50 are logged but not shown to user.
+
+**Standard format:** Use a markdown table for EVERY role type. Columns: Score | Title | Company | Location. Keep it consistent.
+
+**Score breakdown:** For each job, show the 5-factor breakdown (required skills, preferred skills, experience, industry, location) with points awarded. This can be a compact one-line format: "40+20+15+15+10=100" or expanded table — but it MUST be shown.
```

```diff
+8. **Cross-role-type deduplication (MANDATORY before presentation):** Before presenting ANY results, compare filenames (company-slug + title-slug) across ALL `output/verified/*/` directories. If the same filename appears in multiple role-type directories, read only those files to compare scores — keep the copy with the higher score and delete the duplicate. Do NOT skip this step.
```

```diff
 ## SESSION MANAGEMENT

-- **Deterministic checkpoint:** Write session-state.md after every 2 role types reach "verified" status.
+- **Deterministic checkpoint:** Write session-state.md after every 2 role types reach "verified" status. This is a HARD RULE — not a suggestion. After batch 1 completes (2-3 role types verified), IMMEDIATELY write session-state.md before launching batch 2.
```

```diff
+## CONTEXT CONSERVATION
+
+- When dispatching subagents, keep the Task prompt concise. Point subagents to read template files themselves rather than inlining the full template content in the prompt.
+- After presenting a role type's results and receiving user feedback, do NOT re-read verified files. Reference session-state.md summaries only.
+- Digest generation with many hyperlinks: dispatch as a subagent if more than 20 jobs need linking.
```
